浅谈流量领域业务

发表于2024-02-08|更新于2024-02-08
|字数总计:2.9k|阅读时长:8分钟|阅读量:

提前实习已经俩月多了，由于部门是做 C 端用户体验的，在此期间我接触了较多关于流量场的业务，也做了一次相关的业务串讲。本文会基于我的梳理，分享一下我对流量领域业务的浅薄理解。

平台 C 业务理解（流量领域）

流量领域都包括啥

废话不多说，直接上图。这个图是去年入职没多久做的，大概描述了平台 C 流量领域的业务范围，最直观的是大首页，包括球区、腰部 banner 和大首页 feed，然后是我家（归宿项目），此外还包括个人中心、地图找房、综合搜索等业务，你会发现这些都是流量的一级入口，掐断了其中一个，后方就会有大量业务受阻。

流量领域的价值和意义

看了前面的描述，你可能会有些不明觉厉，好像流量场是很高大上的东西，那么它到底是什么东西呢？它又有什么价值呢？

流量场在整个业务流程中位于流量入口，主要负责流量分发相关业务，一方面需要保证代码质量和容错机制，避免因为平台的崩溃而导致大面积业务瘫痪，毕竟入口都没了，又怎么导流你去访问核心业务呢？另一方面，由于贝壳现在启动一体三翼的战略升级，有着多条一赛道和二赛道业务线（二手、新房、家装、惠居等等），作为流量入口，平台这边可以通过调控流量的分发，来提升整体平台流量的商机转化。

流量领域怎么干才能达成这些目的

笔者举一个具体的例子：归宿项目。归宿项目的核心价值就是导流。通过包含用户买房、卖房、租房、装修等完整生命周期的资产卡片，聚合围绕资产的生态，提供一站式的居住生活服务，引导新房二手交易完用户转型追求家装家居服务，提升家装交易的转化率。

归宿

归宿是这边的一个重点项目，这个项目的 idea 真的很好，从设计到落地都值得我学习。

背景

贝壳当前用户认知为【找房工具】，与平台【居住服务】定位不符。用户在 APP 中，并没有灌输关于【居住】的强概念和独立心智。不仅没有足够的内容、权益来承载，相关心智承接的很散落（例如我的房子、报盘、家服等），更是弱化了心智的建立。

“归宿”由此诞生，致力于为用户灌输“居住平台”的理念，围绕资产的生命周期，为资产聚合以其为中心的生态，为报盘、家装等业务导流。

业务结构

归宿整体由资产卡片、管理我家、模块化卡片、订阅、问卷调研等板块组成。从用户的视角来看如下图划分。

资产卡片显示了业主上报的资产信息以及贝壳基于模型得出的房产估价信息。点击资产卡片会进入估价二级页，展示估价因素、同小区价格走势等更为详细的估价信息，有经纪人预约和卖房业务的导流。

管理我家可以对资产进行管理，如更新资产信息、添加我家。

模块化卡片又分为多种，典型的有社区服务、小区房源、市场行情、小区行情、让家变美、让家好住等卡片，这些都是基于选定的某个资产而生成的，就是对该资产的分析和资料补充，同时有让家变美等卡片导流到家装等业务。模块化卡片的业务较为复杂，同时也共存有多个版本，一部分的卡片由平台自己维护，另一些卡片依据给定的规则，由其他第三方业务方提供，平台仅转发透传。

当用户没有添加资产或者是未登录态时，我家的主页的展示不同，会从服务端给一些静态数据引导用户添加资产。

生命周期

登上贝壳 App 的业主用户最关心的莫过于“我家房子能卖多少钱”，在资产卡片中，会标注由贝壳估价模型计算的估价值，满足了一部分业主用户的诉求。而对于交易资产卡片，作为买方用户，关心的是交易进度、交易行情、小区周边配置和如何融入小区（小区生活指南），这些都在其下方的模块化卡片中展现。对于在售业主，也有在售热度、服务动态和房屋管理等特有的卡片帮助业主售房。对于装修中业主，会有历史的施工信息记录、每日进度播报、节点验收报告和工期表等帮助业主掌握装修详情。对于已竣工的业主，会将装修历程浓缩为记忆供业主回顾，还会贴出售后保障的渠道让业主安心。

在后端的实现中，由于对接的是端上和业务方，我们高度强调配置化，对于多种的资产卡片，我们可以通过更改配置来对资产卡片的类型进行上下线开关控制、版本控制、ab 实验编排控制等。

资产生态

资产生态即模块化卡片，也可以叫资产详情页，每切换到一个资产，模块化卡片会跟着变化，下面这些都是模块化卡片。

前置处理

前置处理会过滤一些模块，并处理部分模块 format 时依赖的数据。

配置化模块：首页 feed 和归宿的模块卡片的配置化有点不同，首页 feed 的卡片是不固定的，刷出来有啥卡片都是从推荐接口那边获取，配置化知识作为一层滤网来过滤获取到的卡片。而归宿这边的模块卡片是固定的，配置里有配置啥就会刷出来啥，配置化不仅是过滤，也是模块内容的来源。

如果前端传参带有 scene（对应普通资产和其他不同资产类型），会跟着 scene 去获取对应的配置，当没传 scene 时，就对应无资产&未登录的情况，获取默认的 default_modules 配置，这意味着无资产&未登录情况其实算一种特殊的资产卡片。

工厂实现

模块化卡片主要使用了工厂模式，每个卡片需要各自实现模块数据获取的逻辑，然后根据配置来决定哪些模块可能会被过滤，留下的模块通过工厂获取到对应的实现，而后借助 Golang 并发的特性，对每个模块起一个 goroutine 并行执行，执行完毕后，再统一处理结果。

上图可以看出这个模块包含两个 item，api 访问业务子服务，对模块进行依赖数据的加载、流程的流转和数据聚合。

转发透传模块的实现是 api 会根据设定好的参数模板，填入需要的参数，然后作为客户端去访问业务依赖接口获取数据。顺带一提，转发透传的卡片通过配置化就可以完成上线，不需要平台这边做代码改动，这是因为后方有一个通用处理获取卡片数据、处理卡片 format 的业务子服务。

总体流程

我们可以回顾一下：配置化模块卡片是如何通过配置来控制卡片内容、卡片编排、ab 测试？

通过客户端传参带的 scene 到 apollo 获取对应的模块列表配置（包括模块唯一标识、标题、版本、状态位、权重、ab 实验分组等），然后再根据开城、有效位 status、客户端版本、abgroup 过滤模块，再通过 weight 来排序编排。如果想要去除某个模块，只需要简单地将 status 置为 0 即可，如果想纳入 ab 实验编排，为对应的模块配置添加 abgroup 即可，但是需要保证只有一个在途模块实验，防止实验偏差。

个人成长

实习以来最大的收获是思维上的转变和学会模块拆解。在学校，一切都以学习为主，我不需要去关注我的产出，所以接触的很多东西都是走马观花，学习后可能只留下只有自己看得懂的笔记。而在公司，就需要想想能为公司做出什么贡献，梳理业务后能不能有通俗的文档让组内的人都快速了解这块的业务背景和一些细节，引导后续的同学快速理解这一块业务。

一些在校与在职的区别：

  • 在学校实验室干活时：需求都是老师提的，有些需求可能在开发角度来看不太合理，但也尽量去做，而且通常一人身兼数职，比如数据处理、开发、测试和运维，很容易拖慢了开发效率，而且排期不太明确，今天感觉比较累了，拖到明天再写也可以。

  • 而在这里分工明确、流程规范化、排期时间比较明确，基本不会出现上述的情况，需求评审里会讲不合理的需求打回，排期会定人力、开发时间、提测日期和上线日期，如果其中一环没完成会拖累项目后续的推进。

总结

本篇文章主要分享我对流量领域业务的浅薄理解，以梳理的归宿一部分业务来具体分析，流量领域业务是位于流量入口，主要负责流量分发的业务，指标有 dau、mau、uctr 等，归宿通过聚合业主用户希望看到的数据来提高流量和留存率，通过展示不同业务的卡片来达到导流的目的。流量领域远不止我所说的这么一点，就像归宿项目，还有着 B 面的大头，笔者后续会分享更多关于流量相关的业务。
